# What it is

The Strobes platform runs autonomous pentest agents in the cloud — orchestration, guardrails, and the LLM live remotely. This CLI is the local half: it connects to a thread over a WebSocket and, when the agent needs to touch a real environment, your machine becomes the agent's sandbox.

That means the agent's shell commands run against the workspace files synced to your disk, and its browser tools drive a real Chrome on your screen — while the heavy lifting (planning, models, findings, approvals) stays on the backend.

  • Stream agent runs live — thinking, tool calls, results, Markdown.
  • Run the shell & browser the agent asks for, locally.
  • Manage workspaces, threads, findings, approvals and credits.
  • Multi-tenant, cross-platform, self-updating.

# Install

Prebuilt binaries for every platform are on the releases page. The one-liners detect your architecture, fetch the latest release, and put strobes on your PATH.

macOS / Linux

curl -fsSL https://raw.githubusercontent.com/strobes-co/strobes-agents-cli/main/install.sh | bash

Windows (PowerShell)

irm https://github.com/strobes-co/strobes-agents-cli/releases/latest/download/strobes-x86_64-pc-windows-msvc.tar.gz -OutFile s.tgz
tar -xzf s.tgz; .\strobes-x86_64-pc-windows-msvc\strobes.exe --help

Already installed? Self-update

strobes update           # downloads the latest release and replaces the binary in place

Pick a binary manually

PlatformAsset
macOS (Apple Silicon)strobes-aarch64-apple-darwin.tar.gz
macOS (Intel)strobes-x86_64-apple-darwin.tar.gz
Linux (x86-64)strobes-x86_64-unknown-linux-gnu.tar.gz
Windows (x86-64)strobes-x86_64-pc-windows-msvc.tar.gz

Each ships a .sha256 checksum. Build from source with cargo build --release (needs Rust; Chrome only for the browser tools).

# Quick start

# 1. Authenticate — get a MasterKey from Strobes UI → Organization → API access
strobes login            # interactive (or --base-url / --org-id / --master-key)
strobes status           # ✔ connection OK · ✔ up to date

# 2. Work
strobes                  # open the chat TUI: pick a workspace → thread → chat
strobes workspaces       # list workspaces (+ thread counts + credits)
strobes credits          # AI credit usage, org total + per-workspace

Credentials persist to ~/.config/strobes-ai/config.json (macOS: ~/Library/Application Support/strobes-ai/config.json). Manage multiple deployments as tenants and switch with --tenant <name>.

# Features in depth

Multi-tenant

One CLI, many deployments

Each Strobes deployment is a tenant with its own base URL, org, MasterKey and bindings, all kept in one config.json. The first one you configure becomes the default; switch per-command with --tenant. strobes tenants lists them with the default starred.

strobes --tenant acme login   # first → default
strobes --tenant staging login
strobes tenants
# ★ acme       450d6756…  https://app.strobes.co
#   staging    9f0c…      https://staging.strobes.co
strobes --tenant staging chat
Credits

Live AI credit visibility

Know exactly what a run costs. The status bar shows the ◈ current run and the ⛁ chat lifetime usage live as tokens stream. strobes credits reports org totals plus a per-workspace breakdown, and the workspace list shows credits per workspace.

strobes credits
# ◈ organization:  12.9 credits · 8.2M tokens · 33 runs
# by workspace:
#   3.44 cr · 2.6M tok · 7 runs   Mobile RASP & NPCI Checks
#   3.10 cr · 3.7M tok · 8 runs   SAST & SCA Review
strobes credits --thread <uuid>
Workspaces

Create, bind, and sync

Pick ➕ New workspace in the picker and the CLI prompts for a name and an optional setup prompt, then drops you into the AI setup chat with that prompt sent. Binding a workspace downloads its files into a local sandbox so the agent's execute_command and workspace_get_meta see the real tree. Pickers are type-to-search.

strobes                  # pick ➕ New workspace → name + prompt
strobes bind --new --name "Acme recon" --download
strobes pull --workspace <uuid>   # files → local folder
# in chat:  ^L browse files · ^E open in Finder/Explorer
Local execution

Your machine is the sandbox

When the agent calls execute_command / execute_code, the CLI runs it locally and streams the output back — Python, Bash, Node and Ruby, in the synced workspace dir. On Windows it uses cmd.exe and the right interpreters. The agent never executes on your box without you running the CLI.

# probe the local-execution path headlessly
strobes probe --thread <uuid> --send \
  'run: echo OK && uname -a'
# [probe] execute_command LOCAL  echo OK && uname -a
# [probe] local tool done 11ms exit=0
Browser

A real local browser, driven by the agent

Nine browser_* tools drive a real Chrome via the DevTools Protocol (chromiumoxide). The CLI auto-detects Chrome; if it's missing it returns platform-specific install guidance, honours STROBES_AI_CHROME, or with STROBES_AI_BROWSER_AUTOINSTALL=1 downloads a self-contained Chrome for Testing. Headed by default; STROBES_AI_BROWSER_HEADLESS=1 for headless.

STROBES_AI_BROWSER_HEADLESS=1 strobes probe \
  --thread <uuid> --send \
  'browser_navigate to https://example.com, report the title'
# browser_navigate LOCAL  https://example.com
# tool_output: "Example Domain — https://example.com/"
Resilience

Auto-reconnect & self-update

If the network drops, the WebSocket re-dials with capped backoff instead of sitting idle — the handle and channel survive reconnects, so the UI just shows reconnecting… then connected. The CLI also checks for newer releases on start and in status, and strobes update upgrades in place.

# status bar on a flaky link:
# ○ disconnected — reconnecting in 2s…
# ● connected
strobes status
# ⬆ update available: v0.3.0 — run `strobes update`

⌨ Type-to-search pickers

Workspace/thread pickers filter as you type, bottom-anchored, single alternate-screen (no flicker). Esc navigates back instead of quitting.

⎘ Copy & select

The mouse isn't captured, so native drag-select + copy works in the transcript; ^Y copies the whole conversation.

◐ Working spinner

A spinner in the status bar shows activity during any turn — including long tool / HTTP waits.

⚡ Fast startup

Startup REST + WebSocket round-trips run in parallel; a dev DNS bypass skips slow .local mDNS lookups.

⊞ Context indicator

The transcript shows the bound workspace · current task top-right and the authenticated tenant + version in the banner.

⚑ Cross-platform

macOS (arm64/x64), Linux, and Windows binaries from CI — native shell + browser handling per OS.

# How it works

The remote agent decides; your machine executes its shell & browser tools and streams results back over the pulse WebSocket. Auth is MasterKey (Authorization: token …, WS ?api_key=); workspaces, threads, history, files, findings, approvals and credits come over a MasterKey-authed REST surface.
